# Architecture — Квартирник у Маши Боруха

Документ описывает архитектуру сайта и модель данных, необходимую для разработки CMS.

---

## 1. Текущий стек

| Слой | Технология |
|---|---|
| Хостинг | GitHub Pages (статика) |
| Сборка | Нет — чистый HTML/CSS/JS |
| Шрифты | Google Fonts (Cormorant Garant + DM Sans) |
| Стили | Один файл `styles.css`, CSS-переменные |
| JS | Vanilla, ~20 строк inline в каждой странице (IntersectionObserver + nav toggle) |
| Изображения | Локальная папка `img/`, JPEG/PNG |

Весь контент жёстко зашит в HTML. CMS должна заменить ручное редактирование файлов.

---

## 3. Навигация

Порядок пунктов меню (одинаковый на всех страницах):

```
Главная → Событие → Правила → Плейлист → Архив → Команда → Финансы
```

Активный пункт помечается `aria-current="page"`.

## 4. Дизайн-система

### Цветовые токены (`styles.css`, `:root`)

| Переменная | Hex | Применение |
|---|---|---|
| `--ink` | `#18100A` | Основной текст, тёмный фон |
| `--paper` | `#F3EDE2` | Фон страницы |
| `--wine` | `#5C1A28` | Акценты, кнопки, заголовки |
| `--wine-lt` | `#7D2438` | Hover для wine |
| `--gold` | `#A87828` | Декоративные метки |
| `--smoke` | `#7A6F66` | Второстепенный текст |
| `--surface` | `#EDE6D8` | Карточки, выделенные блоки |
| `--white` | `#FDFAF5` | Белые поверхности |

### Типографика

| Переменная | Шрифт | Применение |
|---|---|---|
| `--f-display` | Cormorant Garant | Все заголовки h1–h3, display-текст, italic акценты |
| `--f-body` | DM Sans | Весь остальной текст |

### Анимации

- Класс `.reveal` → при попадании в viewport добавляется `.visible` (fade + translateY)
- Задержки `.d1`–`.d5` (0.1s–0.6s)
- IntersectionObserver инициализируется inline в каждой странице

## 7. Изображения

### Текущая схема именования

| Паттерн | Описание |
|---|---|
| `perf1.jpg` – `perf5.jpg` | Живые фото выступлений |
| `ev{N}_{1-3}.jpg` | Фото архивного события N (1 = старейшее) |
| `team1.jpg`, `team2.jpg` | Фото команды |

### Для CMS

- Загрузка фото при добавлении нового события → автоматическое именование `ev{N}_{1..3}.jpg`
- При добавлении события в архив — сдвиг индексов не нужен, лучше использовать `ev_{date}_{1..3}.jpg` (например `ev_2025-04-07_1.jpg`)
- Хранилище: достаточно папки `img/` в том же репозитории (GitHub Pages ограничение ~1 ГБ)

## 9. Рекомендуемая архитектура CMS

### Вариант A — минимальный (JSON + шаблоны)

```
cms/
├── data/
│   ├── events.json
│   ├── archive.json
│   ├── playlist.json
│   ├── team.json
│   ├── finance.json
│   ├── rules.json
│   └── homepage.json
├── templates/
│   ├── index.html.jinja
│   ├── events.html.jinja
│   └── ...
├── build.py          — читает JSON + рендерит шаблоны → папка dist/
└── admin/            — простой веб-интерфейс для редактирования JSON
```

**Плюсы:** нет зависимости от БД, всё в git, GitHub Pages продолжает работать.  
**Минусы:** нужен шаг сборки при каждом изменении.

### Вариант B — headless (API + SSG)

- Контент хранится в Notion/Airtable/Supabase
- GitHub Action при изменении данных запускает сборку и деплоит на Pages
- Шаблонизатор: Jinja2 (Python), Nunjucks (Node), или любой SSG (Eleventy, Hugo)

### Вариант C — полноценный бэкенд

- FastAPI / Express бэкенд с SQLite или PostgreSQL
- Панель администратора (например, на базе этого же дизайн-системы)
- Хостинг меняется с GitHub Pages на VPS / Railway / Render

## 10. Приоритеты для CMS

Поля, которые меняются чаще всего и должны быть редактируемыми в первую очередь:

1. **Следующее событие** — дата, место, статус регистрации, ссылка на регистрацию
2. **Архив** — добавление нового события с фотографиями
3. **Команда** — добавление/удаление имён в группах, список организаторов
4. **Плейлист** — добавление/удаление песен
5. **Финансы** — добавление строк расходов
6. **Ссылки** — Telegram, донат, таблица (меняются редко, но все сейчас `#`)
